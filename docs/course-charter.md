# Course Charter

## Mission

Develop the ability to design maintainable enterprise Angular systems, explain the reasoning behind
each decision, and publish verifiable evidence through code, tests, diagrams, and documentation.

## Learning contract

1. No Angular term is assumed knowledge.
2. Every new concept starts with the problem it solves.
3. A small isolated example comes before its enterprise application.
4. Patterns are introduced only when the current requirement creates evidence for them.
5. Alternatives and trade-offs are discussed; no pattern is presented as universally correct.
6. Each chapter ends with working code, tests, documentation, and an architecture review.
7. Public artifacts use clear professional language and never claim experience that the repository
   cannot demonstrate.

## Teaching sequence

```text
Company request
  -> beginner mental model
  -> smallest working implementation
  -> observed engineering pressure
  -> architectural options
  -> justified refactoring
  -> tests and review
  -> documented decision
```

## Definition of done for a chapter

A chapter is complete only when the learner can:

- explain every introduced term in their own words;
- connect the concept to a real business or engineering problem;
- implement the behavior without copying the lesson solution;
- test the important behavior and failure path;
- explain why the selected design fits better than its alternatives;
- update the chapter documentation and architecture record;
- pass build, test, and formatting checks.

## Public portfolio standard

- Commit messages describe intent, not merely changed files.
- README content reflects the repository's real state.
- Architecture diagrams match the implementation.
- Decisions include context, alternatives, consequences, and status.
- LinkedIn updates are prepared only after a demonstrable milestone.
- Secrets, credentials, generated output, and private company data are never committed.
